# LingvaUZ Mobile Setup Completion Design

**Date:** 2026-05-13

## 1. Цель и объем работ
Цель: Подготовить мобильное приложение LingvaUZ к тестированию и интеграции с бекендом, выполнив оставшиеся задачи из списка.
Объем: 
- Выполнить `npm install` в директории mobile/ с предварительной проверкой потенциальных проблем
- Протестировать приложение в Expo Go SDK 55 
- Настроить подключение к бекенду (реальный API URL)
- Добавить необходимые ассеты (иконки, splash-изображения)
После выполнения этих шагов мобильное приложение будет готово к дальнейшей разработке или тестированию с реальными данными.

## 2. Архитектура и компоненты
Текущая архитектура мобильного приложения уже реализована и включает:
- **Навигация:** expo-router v5 с группами маршрутов (auth, tabs) и вложенной навигацией для уроков
- **Управление состоянием:** Zustand хранилища (authStore, lessonStore) с persistence через AsyncStorage
- **Серверное состояние:** TanStack Query кэш с кастомным queryClient
- **API слой:** Axios instance в `lib/api.ts` с токен-интерцептором
- **Компоненты:** Полный набор UI-компонентов для уроков (ProgressBar, LivesBar, AnswerFooter, CompleteScreen, ExerciseRenderer и все 8 типов упражнений)
- **Хуки:** Кастомные хуки (useLesson, useNotifications) для инкапсуляции логики
- **Утилиты:** audio.ts для работы со звуком
- **Константы и типы:** centralized colors.ts и index.ts с типами

**Что нужно сделать:**
1. **npm install** - установить все зависимости из package.json
2. **Проверка подключения к бекенду** - убедиться, что базовый URL в `lib/api.ts` указывает на рабочий бекенд (после его запуска)
3. **Добавление ассетов** - разместить файлы icon.png, splash-icon.png, adaptive-icon.png в соответствующих директориях (обычно в assets/ или через app.json/config плагины)
4. **Тестирование в Expo Go** - запустить `expo start` и проверить работу через сканирование QR-кода

Архитектурных изменений не требуется - нужно лишь завершить настройку и проверить работу существующей структуры.

## 3. Поток данных и обработка ошибок
Текущий поток данных в приложении:
- **Авторизация:** Через authStore (Zustand) с хранением токена в AsyncStorage. При входе/регистрации токен сохраняется и используется в Axios интерцепторе.
- **Запросы к данным:** Через TanStack Query с кастомным queryClient. Хуки типа useLesson используют query.fetchAndGet для загрузки данных урока.
- **Локальное состояние:** Уроками управляет lessonStore (Zustand) для хранения текущего состояния урока, прогресса и т.д.
- **Оповещения:** Хук useNotifications обрабатывает streak-напоминания.

**Что нужно проверить/настроить:**
1. **API URL:** Убедиться, что в `lib/api.ts` правильно указан базовый URL бекенда (после его запуска). Сейчас, вероятно, стоит заглушка или localhost.
2. **Обработка ошибок:** Текущая реализация включает:
   - Axios интерцептор для автоматического добавления токена
   - Potentially глобальную обработку ошибок в queryClient (нужно проверить в lib/queryClient.ts)
   - Локальную обработку в компонентах (например, попытки повторного запроса)
3. **Первичная загрузка данных:** При запуске приложения должно происходить восстановление сессии из authStore и проверка авторизации.

После `npm install` и настройки API URL поток данных должен работать как задумано, предполагая, что бекенд запущен и доступен.

## 6. Возможные риски и способы mitigation
- **Риск:** Конфликты версий зависимостей при npm install
  - **Mitigation:** Внимательно читать вывод npm install, обращать внимание на предупреждения о peer dependencies
  
- **Риск:** Бекенд недоступен или работает некорректно
  - **Mitigation:** Сначала проверить работу приложения в офлайн режиме/с моковыми данными, затем подключать к бекенду
  
- **Риск:** Проблемы с Expo SDK версии 55 (несовместимость некоторых зависимостей)
  - **Mitigation:** Проверить логов Expo при запуске, обращаться к документации Expo SDK 55 при необходимости
  
- **Риск:** Ассеты имеют неправильный формат или размеры
  - **Mitigation:** Следовать рекомендациям Expo для размеров и форматов иконок (1024x1024 для адаптивной иконки, etc.)
